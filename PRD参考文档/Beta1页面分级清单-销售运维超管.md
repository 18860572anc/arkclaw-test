| 覆盖角色 | 官方管理员、交付运维、销售 |
| 会议依据 | `20260429102613-arkclaw管理后台首轮评审会议-逐字稿文本-1.txt`、`20260429183822-arkclaw实现讨论&规划-逐字稿文本-1.txt` |

## 一、会议结论提炼

1. **Beta1 先跑起来**：会议明确提到“现在是 beta1 版本先跑起来”，不要一次性把所有后台能力都塞进 15 天。
2. **销售 + 客户多租户是 T0**：会议明确说“别搞到 5 月 20 号之后还没有销售客户的多租户，这两个是必上的 T0 版本”。
3. **官方后台是新做且必须支撑主流程**：客户、订单、对公、开通任务、销售归属、佣金这些都需要在我方官方后台闭环。
4. **运维需要进入 Beta1，但只保留工单工作台**：会议中提到 IM 绑定、网络配置、客户诊断与支持、健康巡检都属于运维能力；考虑 15 天周期，Beta1 不再单独建设这些运维页面，而是在工单详情中跳转客户侧已有页面处理。
5. **Beta1 支付只做对公**：会议明确“企业买一定是公对公转账”，不接微信 / 支付宝充值。
6. **火山映射属于技术集成，不应变成前台复杂页面**：前端原型只表达状态和异常，不把火山账号、空间、Coding Plan 的底层细节暴露给客户或销售。
7. **官方后台 Beta1 不拆子角色**：R1.0 超管 / R1.1 运营 / R1.2 财务 / R1.3 客服先不做产品区分，统一为“官方管理员”。

## 二、Beta1 主流程定义

Beta1 必须能跑通这一条：

```text
销售生成分享链接 / 手动录入客户
  -> 客户归属到销售
  -> 客户注册企业 / 进入企业空间
  -> 客户生成对公订单并提交付款凭证
  -> 官方管理员审核对公到账
  -> 系统生成企业开通任务
  -> 运维接收开通交付工单
  -> 运维从工单详情进入客户侧页面协助处理
  -> 官方后台 / 运维同步火山资源映射、SSO、席位池状态
  -> 客户管理员分配席位
  -> 成员登录后席位生效
  -> 销售看到开通完成和佣金 pending
```

Beta1 不追求：

- 不追求完整运维交付平台，但必须保留工单流转和客户侧协助入口。
- 不追求复杂佣金政策配置。
- 不追求销售团队管理和排行榜。
- 不追求支付通道、自动银行流水识别。
- 不追求资讯、系统参数、监控告警、全量权限矩阵等平台后台能力。

## 三、版本分级口径

| 优先级 | 含义 | 15 天内是否必须 |
|---|---|---|
| T0 / Beta1 必做 | 没有它主流程无法闭环 | 必须做 |
| P1 / Beta1.1 | 不影响主链路，但影响运营效率或异常处理 | 可在 Beta1 后 1-2 周补 |
| P2 / Beta2 | 管理增强、精细化运营、风控、规模化能力 | 后置 |
| P3 / 暂缓 | 有原型价值，但短期不该压研发 | 暂不进开发范围 |

## 六、交付运维页面分级

### 6.1 Beta1 必做原则

运维需要进入 Beta1，但范围必须收敛成“工单流转 + 客户侧代入入口”。

Beta1 运维不做完整平台，不重复建设客户侧已有配置页，只解决四件事：

1. **接单**：对公审核通过 / 开通任务生成后，运维能看到交付工单。
2. **定位**：工单详情能看到客户、订单、开通任务、当前卡点和处理记录。
3. **进入客户侧**：运维从工单详情跳转到客户侧席位、用户、网络、工作台、可观测、审计、安全等页面协助处理。
4. **回填**：运维处理完成后，在工单中记录结果，并回填到开通任务，让销售、客户、官方管理员看到同一状态。

### 6.2 Beta1 必做

| 页面 | 路由 | 必做原因 | Beta1 做到什么程度 |
|---|---|---|---|
| 交付工单队列 | `/ops/tickets` | 运维需要接收开通任务、开通异常和销售反馈，否则所有交付压力会堆在官方后台 | 做工单列表、优先级、状态、负责人、来源、SLA |
| 工单详情 | `/ops/tickets/:id` | 运维必须在一个页面完成处理记录、状态推进、客户侧跳转和结果回填 | 做工单信息、客户信息、处理备注、客户侧快捷入口、完成 / 暂挂 / 转交 |

## 八、必须砍掉的范围

为了保证 15 天内能交付，以下内容不应进入 Beta1：

| 砍掉项 | 原因 |
|---|---|
| 完整运维平台 | Beta1 只做开通交付最小闭环；巡检、知识库、自动化编排后置 |
| 运维独立 IM/SSO、网络、诊断页面 | 客户侧已有对应能力，Beta1 从工单详情跳转客户侧承接 |
| 微信 / 支付宝支付 | 会议确认 Beta1 企业购买走对公 |
| 复杂佣金政策配置 | Beta1 保留基础佣金规则；多级政策、申诉、批次发放后置 |
| 销售团队排行榜 / 经理视角 | 管理增强，不影响单销售闭环 |
| 系统配置、字典、通知模板 | 可以先硬编码核心通知和状态 |
| 新闻资讯后台 | 与购买开通主流程无关 |
| 代登录 | 高风险能力，不适合 Beta1 rushed 上线 |
| 健康巡检和告警规则 | 依赖监控数据，后置 |
| 火山底层映射精细控制页面 | Beta1 只展示状态，真实能力由技术对接 |

## 九、验收口径

Beta1 页面验收不按“页面数量”验收，而按流程验收：

| 验收问题 | 必须答案 |
|---|---|
| 销售能否让客户归属到自己名下？ | 能，通过分享链接 / 邀请码 / 手动录入 |
| 客户能否完成对公订单提交？ | 能，生成订单、备注号、上传凭证 |
| 官方后台能否审核对公？ | 能，通过 / 驳回 / 挂起 |
| 审核通过后是否能生成开通任务？ | 能 |
| 开通任务是否能流转到席位池可用？ | 能 |
| 客户管理员是否能分配席位？ | 能 |
| 成员是否有进入工作台并启用席位的承接？ | 能 |
| 销售是否能看到开通完成和佣金 pending？ | 能 |
| 运维是否必须介入主流程？ | 必须有最小闭环：接收工单、进入客户侧协助处理、记录结果、回填开通任务 |

## 十、最终建议

Beta1 不要按“我们现在原型里有多少页面”来开发，而要按“主流程必须经过哪些页面”来开发。

建议最终范围：

```text
必须做：
销售 7 页
官方管理员 14 页
运维 2 页
```
