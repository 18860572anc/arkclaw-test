## 0. 文档说明

### 0.1 文档目的

本文档是ArkClaw Beta1版本的完整功能测试覆盖指南，旨在：

1. **QA测试用例编写**：提供370+个详细测试用例，包含具体操作步骤、输入数据、预期结果和验收标准
2. **产品需求验证**：对照PRD文档确保所有Beta1需求点都有对应测试覆盖
3. **开发自测清单**：为开发人员提供功能模块的自测检查点和关键流程验证依据

### 0.2 术语定义

| 术语 | 定义 |
|------|------|
| USCC | 统一社会信用代码（Unified Social Credit Code），企业唯一标识 |
| E2E | End-to-End，端到端测试，覆盖完整业务流程 |
| Mock | 模拟数据/接口，Beta1阶段无真实后端，所有API返回Mock数据 |
| T0必做 | Beta1必须完成的核心功能，缺失则主流程无法闭环 |
| P0/P1/P2 | 测试用例优先级：P0阻断性、P1核心功能、P2增强功能 |
| 归属期 | 客户首次可用起12个月，期间销售可获得5%固定佣金 |
| 开通任务 | 对公审核通过后系统自动生成的企业资源开通流程 |
| 席位池 | 企业购买的席位总数，由管理员分配给成员使用 |

## 1. 测试策略总览

### 1.1 Beta1 核心业务流程图

```
销售生成分享链接/邀请码
    ↓
客户通过链接注册企业账号（USCC去重校验）
    ↓
销售归属自动绑定
    ↓
客户选择套餐 → 生成对公订单 → 上传付款凭证
    ↓
官方管理员审核对公到账（通过/驳回/挂起）
    ↓
系统自动生成开通任务
    ↓
运维接收工单 → 跳转客户侧协助处理 → 回填开通状态
    ↓
火山资源映射 + SSO配置 + 席位池开通
    ↓
客户管理员进入席位管理 → 分配席位给成员
    ↓
成员登录工作台 → 席位自动启用
    ↓
销售看到佣金pending → 财务T+1确认后转confirmed
```

### 1.2 测试范围界定

**Beta1 T0必做页面（23页）**：

| 角色域 | 页面数量 | 页面列表 |
|--------|---------|---------|
| 销售侧 | 7页 | 工作台、客户池、新增客户、客户详情、分享链接、开通跟进、佣金明细 |
| 官方管理员侧 | 14页 | 仪表盘、企业管理、企业详情、订单总览、对公审核、开通任务、退款处理、销售管理、销售详情、留资管理、佣金规则、佣金流水、套餐价格、内部员工账号 |
| 运维侧 | 2页 | 交付工单队列、工单详情 |
| 客户侧主链路 | - | 注册页、对公购买页、席位管理页、工作台页 |

**后置范围（不进入Beta1测试）**：
- 微信/支付宝支付（Beta1仅对公转账）
- 完整运维平台（IM绑定、网络配置、诊断、巡检独立页面）
- 销售排行榜、团队管理
- 复杂佣金政策配置、申诉审批流
- 系统监控告警、字典管理、通知模板
- 资讯管理、代登录等高风险能力

### 1.3 测试类型矩阵

| 测试类型 | 占比 | 说明 |
|---------|------|------|

### 1.4 Mock数据依赖说明

Beta1阶段所有API均为Mock数据，测试前需确认：

1. **Mock服务运行正常**：`frontend/src/services/*MockApi.ts` 文件已加载
2. **预设测试数据可用**：销售账号、客户企业、订单、工单等Mock数据已初始化
3. **Mock开关可配置**：能够灵活切换特定接口的成功/失败响应
4. **火山API Mock到位**：CreateSpace、AllocateSeats等关键接口有Mock返回

详细Mock数据模板见第7章。

## 8. 验收标准与通过准则

### 8.1 主流程验收Checklist（必须100%通过）

- [ ] 销售能否让客户归属到自己名下？（分享链接/邀请码/手动录入）
- [ ] 客户能否完成对公订单提交？（生成订单、备注号、上传凭证）
- [ ] 官方后台能否审核对公？（通过/驳回/挂起）
- [ ] 审核通过后是否能生成开通任务？
- [ ] 开通任务是否能流转到席位池可用？
- [ ] 客户管理员是否能分配席位？
- [ ] 成员是否有进入工作台并启用席位的承接？
- [ ] 销售是否能看到开通完成和佣金pending？
- [ ] 运维是否必须介入主流程？（接收工单、进入客户侧协助、记录结果、回填开通任务）

### 8.2 性能指标要求

| 指标 | 目标值 | 测试方法 |
|------|--------|---------|
| 登录后首屏加载 | ≤2s (P95) | Lighthouse + 网络节流模拟 |
| 列表页查询 | ≤1s (P95) | Mock延迟注入测试 |
| 分享链接批量派发（50员工×1模板） | ≤30s | 压力测试 |
| 批量导入500员工 | ≤60s | 大数据集Mock |
| 火山API调用成功率 | ≥99.5% | Mock失败率注入 |

### 8.3 缺陷等级定义

| 等级 | 定义 | 示例 | 发布影响 |
|------|------|------|---------|
| **Blocker** | 主流程阻断，无法继续测试 | USCC去重失效导致重复客户 | 禁止发布 |
| **Critical** | 核心功能错误，影响业务闭环 | 对公审核通过后未生成开通任务 | 禁止发布 |
| **Major** | 功能可用但有明显缺陷 | 佣金计算误差>5% | 修复后发布 |
| **Minor** | 体验问题或非关键路径缺陷 | 响应式布局轻微错位 | 可带病发布，P1修复 |
| **Trivial** | 文案/样式微调 | 按钮hover色值偏差 | 可带病发布 |

### 8.4 Beta1发布准入条件

1. 所有P0/P1测试用例通过率100%
2. 无Blocker/Critical级别缺陷
3. Major级别缺陷≤3个且有明确修复计划
4. 主流程E2E测试全部通过（第2章）
5. 权限隔离测试全部通过（第4章）
6. UI规范检查无严重违规（第5章）
7. 性能指标达标率≥90%

### B. 高风险测试点

| 风险项 | 影响 | 应对策略 |
|--------|------|---------|
| Mock数据与真实API差异 | 上线后可能出现未测出的bug | 预留API联调回归测试窗口 |
| 火山API能力不足 | 某些页面无法实现PRD定义的功能 | 提前对齐API清单，无API能力以"请联系客户经理"过渡 |
| 并发去重失效 | 客户被多个销售重复录入 | 数据库UNIQUE约束兜底 + 应用层乐观锁 |
| 佣金计算精度 | 财务对账偏差引发销售投诉 | calc_basis_json永久存档，提供差异查询工具 |
| 企业激活入口外泄 | 非管理员尝试确认企业订单 | 入口绑定tenant + order + admin权限校验 |

### C. 测试执行建议顺序

1. 先跑通E2E主流程（第2章），确保主链路可用
2. 再按角色分域测试（第3章），覆盖各页面功能
3. 然后执行权限隔离测试（第4章），确保数据安全
4. 接着进行UI规范检查（第5章），保证视觉一致性
5. 最后执行异常流程测试（第6章），验证边界条件
