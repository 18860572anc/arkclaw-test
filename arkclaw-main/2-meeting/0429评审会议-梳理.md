## 三、关键新增 / 变更（**需要回写 PRD**）

### 🔴 1. 邀请码 → **激活码**（语义重大调整）

**原 PRD 设计**：邀请码用于注册归因（与分享链接并列）。

**会议重新定位**：邀请码 = 激活码 = 确认码（同一概念），用作**席位激活凭证**。

**两种使用模式**：

**模式 A · 客户自助驱动**
```
客户自己在官网 → 注册 → 自助购买套餐 → 我方后台审核
   ↓
后台分配客户经理（销售） → 客户经理给客户激活码
   ↓
客户输入激活码 → 席位激活 → 开始使用
```

**模式 B · 销售驱动**
```
客户注册（空账号停留在那里）
   ↓
销售认领（公共池） / 销售已绑定客户
   ↓
销售推动客户购买 → 客户付款
   ↓
后台生成激活码 → 销售给客户 → 客户激活
```

**核心特点**：
- 后台可**批量生成 1000 个激活码**
- 激活码可承载**政策**（绑定特定套餐 / 折扣 / 试用期）
- 客户能自主完成"注册 + 购买"两步，无需我方分配账号
- 我方负责"审核 + 发激活码"两步
- 参考产品：Manus 早期邀请制

> ⚠️ **注意**：与现有 PRD 的"分享链接归因"是**两个独立机制**，不可混淆。
> - 分享链接 = 归因层（客户注册时绑定销售）
> - 激活码 = 激活层（客户购买后激活席位）
