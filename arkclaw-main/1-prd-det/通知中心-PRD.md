- **客户成员**：本期不再作为产品角色存在，不纳入本 PRD 范围。

## 二、模块定位、目标与设计原则

### 2.1 模块定位

通知中心是 ArkClaw 代理平台中各业务角色共用的一层基础能力，用来承接各业务模块的“系统告诉用户下一步该做什么”。

它不负责生成业务动作，也不负责模板治理本身，而是负责：

1. 将关键业务事件汇总到统一入口。
2. 让用户在不离开当前工作台的情况下快速感知未读事项。
3. 提供明确跳转，把用户带回对应业务页继续处理。

### 2.2 本期目标

1. 解决当前顶部铃铛不可点击、不可查看通知的问题。
2. 为客户管理员、代理、交付运维、交付管理员、财务、官方管理员提供统一的站内通知收件入口。
3. 建立“顶部铃铛 + 最近通知面板 + 完整通知页”的一套稳定交互。
4. 打通“业务事件触发通知”“通知模板生成文案”“通知偏好控制接收”三者边界。

### 2.3 设计原则

1. **先解决收件问题，再扩展发送平台**：本期只细化站内通知中心，不展开邮件、短信发送基础设施。
2. **通知必须可行动**：每条通知都必须有明确业务目标页，不能出现“看到了但无处可去”的死消息。
3. **按角色裁剪内容，不做统一大杂烩**：各角色共享框架，但通知类型、标题文案和跳转不同。
4. **未读数必须可信**：铃铛红点、面板未读、完整页未读三者状态严格一致。
5. **首版不追求复杂治理**：不做删除、收藏、分组订阅、通知撤回、跨渠道投递监控。

## 四、信息架构与入口设计

### 4.1 顶部入口

各角色工作台顶部统一保留铃铛按钮。

- 客户管理员：`/tenant/*`
- 代理：`/sales/*`
- 交付运维：`/ops/*`
- 交付管理员：`/ops-admin/*`
- 财务：`/finance/*`
- 官方管理员：`/admin/*`

铃铛状态规则：

- 有未读：显示红点和未读数
- 无未读：仅显示铃铛图标，不显示红点
- 点击铃铛：打开最近通知面板

### 4.2 最近通知面板

面板为轻量查看层，不替代完整通知页。

固定结构：

1. 面板标题：`通知`
2. 右上操作：`全部已读`
3. 列表区：最近通知
4. 底部操作：`查看全部`

单条通知卡片字段：

- 通知类型
- 是否未读
- 标题
- 摘要
- 发生时间
- 跳转按钮或整卡点击

面板行为规则：

- 默认展示最近 8 条
- 按时间倒序
- 点击单条：先标记已读，再跳对应目标页
- 点击`全部已读`：当前用户所有未读站内通知变已读
- 点击`查看全部`：进入对应角色的完整通知中心页

### 4.3 完整通知中心页

完整页按角色分路由，不使用公共混合页。

路由固定为：

- 客户管理员：`/tenant/notifications`
- 代理：`/sales/notifications`
- 交付运维：`/ops/notifications`
- 交付管理员：`/ops-admin/notifications`
- 财务：`/finance/notifications`
- 官方管理员：`/admin/notifications`

页面固定结构：

1. 页面标题
2. 筛选区
3. 列表区
4. 批量操作区

筛选区固定能力：

- 状态：全部 / 未读 / 待处理
- 类型：订单 / 审核 / 开通交付 / 留资客户 / 佣金结算 / 系统维护安全
- 时间排序：默认倒序

批量操作固定能力：

- 批量标记已读

首版不提供：

- 批量删除
- 批量归档
- 自定义筛选保存

## 六、详细需求

### 6.1 通知分类

首版统一使用 6 个一级分类：

| 分类 | 说明 |
|---|---|
| 订单 | 下单、支付成功、付款凭证上传、订单取消 |
| 审核 | 对公审核、退款审核、财务复核 |
| 开通交付 | 开通任务、企业可用、交付工单流转 |
| 留资客户 | 留资分配、客户归属变更 |
| 佣金结算 | 佣金到账、佣金结算处理 |
| 系统维护安全 | 系统维护、支持工单、新企业注册 |

### 6.2 通知优先级

首版定义 3 档优先级：

- `high`：必须尽快处理，默认落入“待处理”视图
- `medium`：重要提醒，建议处理
- `low`：信息提示，了解即可

优先级只影响展示样式和“待处理”集合，不额外引入 SLA 机制。

### 6.3 六类角色通知矩阵

#### 客户管理员

首版必须覆盖：

- 标准订单支付成功
- 对公转账审核通过
- 对公转账审核驳回
- ArkClaw 开通完成
- ArkClaw 开通失败
- 系统维护通知

对应默认跳转：

- 订单相关 -> `/tenant/billing`
- 开通完成 -> `/tenant/overview`
- 开通失败 -> `/tenant/consult`
- 系统维护 -> `/tenant/notifications`

#### 代理

首版必须覆盖：

- 留资分配
- 客户付款成功
- ArkClaw 开通完成
- 佣金已到账
- 客户归属变更

对应默认跳转：

- 留资 -> `/sales/leads`
- 客户付款 / 开通完成 / 归属变更 -> `/sales/customers`
- 佣金 -> `/sales/commission`

#### 交付运维 / 交付管理员

首版必须覆盖：

- 被分配到的交付工单，即新开 ArkClaw
- 被分配到的支持工单，即绑定客户提交的技术支持工单
- 被分配到的对公流水号录入任务，即用户提交对公审核后的第一步处理

对应默认跳转：

- 交付运维交付工单 -> `/ops/delivery-tickets/:id`
- 交付运维支持工单 -> `/ops/support-tickets/:id`
- 交付运维对公流水录入 -> `/ops/orders/bank-transfer-review`
- 交付管理员交付工单 -> `/ops-admin/delivery-tickets/:id`
- 交付管理员支持工单 -> `/ops-admin/support-tickets/:id`
- 交付管理员对公流水录入 -> `/ops-admin/orders/bank-transfer-review`

说明：

- 交付管理员与交付运维接收同一类通知。
- 交付管理员额外具备改派能力，但不为改派能力单独增加通知类型。

#### 财务

首版必须覆盖：

- 对公审核待财务复核
- 退款待处理
- 发票申请待处理
- 佣金结算待处理

对应默认跳转：

- 对公复核 -> `/finance/orders/bank-transfer-review`
- 退款 -> `/finance/orders`
- 发票 -> `/finance/orders/invoices`
- 佣金结算 -> `/finance/commission/transactions`

#### 官方管理员

首版必须覆盖：

- 新企业注册

对应默认跳转：

- 新企业注册 -> `/admin/tenants`

### 6.4 单条通知展示规则

每条通知必须具备以下字段：

- `id`
- `category`
- `priority`
- `title`
- `summary`
- `createdAt`
- `read`
- `actionUrl`
- `actionLabel`

文案规则：

- `title` 用一句话表达发生了什么
- `summary` 用一句话说明原因或下一步动作
- `actionLabel` 固定为业务动作，如：`查看订单`、`进入工单`、`查看佣金`

### 6.5 已读与未读规则

- 新创建通知默认 `read=false`
- 用户点击通知卡片后，先标记已读，再跳转
- 用户点击`全部已读`后，将当前账号所有未读站内通知置为已读
- 已读不会删除历史记录
- 未读数只统计 `read=false`

### 6.6 待处理规则

完整页中的“待处理”不是单独状态字段，而是由通知类型和优先级推导出的视图。

首版纳入“待处理”的通知包括：

- 留资待认领
- 交付工单待处理
- 支持工单待处理
- 对公流水号待录入
- 对公审核待财务复核
- 退款待处理
- 发票申请待处理
- 佣金结算待处理
- 开通失败待排查
- 对公驳回待客户补传

信息类通知不进入“待处理”：

- 系统维护公告
- 佣金到账确认
- 订单支付成功
- 新企业注册

### 6.7 与个人通知偏好的关系

个人通知偏好本期保留为用户侧配置能力，不在顶部通知中心中编辑。

首版默认：

- 代理个人中心已存在的通知设置继续生效
- 其他角色文档层面预留统一偏好模型

统一偏好项建议沉淀为：

- `leadAssigned`
- `customerPaid`
- `commissionArrived`
- `maintenance`
- `openingTask`
- `securityAlert`

偏好作用规则：

- 关闭某类偏好后，该类通知不再以“主动提醒”进入个人通知中心
- 强制通知类事件不受偏好影响：对公驳回、企业开通失败、对公流水号录入、财务复核

### 6.8 与通知模板的关系

通知模板页解决“内容怎么写”，通知中心解决“消息给谁看、在哪里看、点进去做什么”。

本期要求在 PRD 中明确三层边界：

1. 模板：`/admin/system/notify-templates`
2. 通知中心：各角色自己的 `/notifications`
3. 偏好设置：个人中心中的通知设置

首版不要求：

- 在通知中心内编辑模板
- 在模板页预览通知中心效果
- 建立复杂模板版本管理

### 7.2 首版接口

#### 通用接口

- `GET /api/notifications/unread-count`

- `GET /api/notifications`

- `POST /api/notifications/:id/read`

- `POST /api/notifications/read-all`
  - 当前账号全部已读

#### 偏好接口

- `GET /api/notification-preferences/me`
- `PUT /api/notification-preferences/me`

- `/sales/notifications`
- `/ops/notifications`
- `/admin/notifications`

顶部铃铛不直接跳页，而是先打开最近通知面板。

## 八、页面与状态要求

### 8.1 空状态

最近通知面板空状态：

- 文案：`暂无通知`
- 不显示红点
- 仍保留`查看全部`

完整页空状态：

- 文案：`暂无通知记录`
- 支持用户清空筛选后回看全部

### 8.2 异常状态

接口失败时：

- 面板内显示轻量错误态：`通知加载失败，请重试`
- 完整页支持刷新按钮
- 未读数接口失败时，不阻塞顶部其他 UI；铃铛默认不显示数字，仅显示图标

### 8.3 性能要求

- 顶部未读数首屏返回目标：`< 500ms`
- 最近通知面板展开后目标：`< 800ms`
- 完整通知页分页查询目标：`< 1s`

## 十、验收标准

### 10.1 功能验收

- 顶部铃铛可点击，点击后可打开最近通知面板
- 面板能展示最近通知、单条已读、全部已读、查看全部
- 六类角色都存在完整通知页，并能展示各自通知
- 通知点击后能跳回正确业务页
- 铃铛未读数、面板未读、完整页未读状态一致
- 代理个人中心现有通知设置与通知中心边界明确，不冲突
- 官方管理员通知模板仍保留独立入口，不与通知中心混淆

### 10.2 业务场景验收

1. 客户管理员上传对公凭证后被驳回
   - 收到站内通知
   - 铃铛出现未读
   - 点击后跳账单 / 订单页补传

2. 代理被分配新留资
   - 收到站内通知
   - 点击后进入 `/sales/leads`

3. 交付运维收到交付工单
   - 收到站内通知
   - 点击后进入 `/ops/delivery-tickets/:id`

4. 交付运维收到对公流水号录入任务
   - 收到站内通知
   - 点击后进入 `/ops/orders/bank-transfer-review`

5. 财务收到对公审核复核任务
   - 收到站内通知
   - 点击后进入 `/finance/orders/bank-transfer-review`

6. 官方管理员看到新企业注册
   - 收到站内通知
   - 点击后进入 `/admin/tenants`

### 10.3 非功能验收

- 空状态、错误状态、无未读状态都有清晰 UI
- 不存在无法点击、点击无反馈、跳转到空白页的情况
- 客户成员角色不会在任何通知 PRD 或 UI 文案中重新出现

## 十一、实施建议

### 11.1 开发顺序

1. 先做顶部铃铛未读数和最近通知面板
2. 再补六类角色完整通知页
3. 最后接业务触发矩阵和个人偏好

### 11.2 Beta1 建议

- 站内通知内容可以先由 mock 数据和关键触发场景驱动
- 邮件 / 短信模板仅保留后台编辑和 trigger 对齐，不要求本轮打通真实发送链路
- 强制通知类事件优先打通：对公驳回、开通失败、留资分配、交付工单、支持工单、对公流水号录入、财务复核

## 十二、默认决策

1. 本 PRD 落盘目录使用仓库现状 `/1-prd-det`，不新建 `prd-det` 平级目录。
2. 角色术语统一使用 `代理`，不再使用对外产品文案 `销售`。
3. 顶部铃铛采用“最近通知面板 + 完整通知页”的两层结构。
4. 首版只定义站内通知中心，不扩展为完整消息中台。
5. 客户成员角色不纳入通知体系。
