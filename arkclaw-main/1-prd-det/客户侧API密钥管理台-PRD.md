## 一、会议结论提炼

### 1.1 核心结论

本需求本质是：在**客户管理员侧**提供一个 API Key 管理台，让每个客户企业可以自己创建多个 Key，并把 Key 分配给不同项目、团队或人员使用。我方平台通过 Key 识别调用方，承接接口转发、用量计量和余额扣费。

会议中明确的业务口径：

1. **不能免费提供外部接口**：外呼、非大模型能力、第三方模型等调用会产生我方成本，必须按客户扣费。
2. **客户预付费后消耗**：客户先充值 / 预付，接口调用从企业余额中扣除。
3. **按 Key 区分调用来源**：每个企业可创建多个 Key；平台接到请求后按 Key 识别企业、分组和调用人。
4. **第一期先简单**：计费复杂时先按次计费；如果外呼接口能稳定返回通话时长，再支持按分钟计费。
5. **第一期重点支撑外呼**：小龙虾招聘自动化场景最后一步需要调用我方外呼接口，外呼调用需要单独计费。
6. **形态参考“柏拉图”式 API 密钥台**：页面类似用户截图中的 API 密钥列表、创建密钥弹窗、使用记录、渠道状态。
7. **转发逻辑先保持简单**：我方网关验证 Key 后，将请求原样转发到实际服务供应商或我方内部接口，再根据结果记账。

### 1.2 与主 PRD 的边界变化

主 PRD 原 § 二曾将“开放平台 / 让客户调我们的 API”列为非目标。本 PRD 是基于 2026-05-21 会议新增的版本决策，建议将其从“非目标”调整为：

| 范围 | 口径 |
|---|---|
| 客户直接调用火山 ArkClaw 原生 API | 仍不做，客户不直接拿火山凭证 |
| 客户调用我方增值接口 / 转发接口 | 本 PRD 新增，统一走我方 API Gateway 和企业 API Key |
| 火山底层凭证、第三方模型 Key | 不暴露给客户，只由我方平台托管 |

## 二、模块定位、目标与范围

### 2.1 模块定位

客户侧 API 密钥管理台是客户企业管理员的**接口消费控制台**，承载：

```text
客户管理员充值 / 拥有可用余额
  -> 创建一个或多个 API Key
  -> 给 Key 设置名称、分组、额度、速率、有效期、IP 限制
  -> 开发者 / 自动化 Skill 使用 Key 调用我方 API Endpoint
  -> 我方网关鉴权、限流、转发到实际供应商 / 内部服务
  -> 记录请求、响应、费用、余额扣减
  -> 客户管理员查看用量、流水、异常和余额提醒
```

它不是客户业务操作页面，也不是火山凭证管理页。客户看到的是我方提供的可调用服务目录和自己的消费记录；底层由我方决定调用火山、火山外呼、第三方模型或自研服务。

### 2.2 本期目标（In Scope）

| # | 模块 | 核心能力 |
|---|---|---|
| A10.1 | API 密钥管理 | 创建、查看、复制、禁用、编辑、删除 Key；Key 仅创建后显示一次明文 |
| A10.2 | 分组与归属 | Key 可选择分组，用于区分项目、团队、员工或自动化任务 |
| A10.3 | 安全限制 | IP 白名单、额度限制、速率限制、有效期、禁用状态 |
| A10.4 | 接口转发 | 统一 Endpoint 接收请求，按服务类型转发到外呼 / 模型 / 多媒体等后端服务 |
| A10.5 | 计量扣费 | 按次计费为第一期默认；外呼支持按分钟计费预留；每次调用生成费用流水 |
| A10.6 | 余额与欠费控制 | 调用前校验企业余额；余额不足拒绝或进入保护策略；支持低余额提醒 |
| A10.7 | 使用记录 | 按 Key、服务、状态、时间、费用查询调用明细 |
| A10.8 | 渠道 / 服务状态 | 展示可用服务、单价、状态、限流、文档入口 |
| A10.9 | 审计与通知 | Key 创建 / 删除 / 禁用 / 高额消费 / 异常失败通知客户管理员 |

### 2.3 本期非目标（Out of Scope）

| 非目标 | 说明 |
|---|---|
| 客户自带供应商 Key | V2 再考虑；本期客户只使用我方生成的 Key |
| 客户直接配置火山 / 第三方原始凭证 | 不开放，避免泄露供应商凭证和成本口径 |
| 复杂模型市场 | 本期只做服务目录，不做自由上架、评分、公开市场 |
| 真实钱包 / 三方支付自动扣款 | 沿用现有账单与充值体系；本期可以是数字余额 + 对公审核 |
| 分账给客户下属人员 | 本期费用只从企业账户扣，不做员工个人钱包 |
| 多级经销接口转售 | 本 PRD 面向最终客户管理员，不处理代理再分发 |
| 完整开发者门户 | 本期提供必要文档、Endpoint、示例；SDK、Playground 后置 |

### 3.1 角色定义

| 角色 | 权限范围 |
|---|---|
| 客户管理员（Tenant Admin） | 全量管理本企业 API Key、分组、限制、使用记录、余额提醒 |
| 客户财务 / 账单管理员（Tenant Finance，可选） | 查看用量、费用、订单、充值；不能创建或查看 Key |
| 客户开发者 / 成员（Tenant Developer，V2） | 可查看被授权的 Key 文档和调用记录；本期可不单独拆角色 |
| 官方管理员 | 可查看企业 API 消费摘要、失败异常、扣费流水；不能查看 Key 明文 |
| 交付运维 | 可查看接口健康与失败日志；不能查看 Key 明文，不能代客户创建 Key |

| 复制 Key | 仅创建后一次 / 已授权时重置 | ❌ | ❌ | ❌ | ❌ |
| 编辑名称 / 分组 / 限制 | ✅ | ❌ | ❌ | ❌ | ❌ |
| 禁用 / 启用 Key | ✅ | ❌ | ❌ | 仅紧急封禁 | ❌ |
| 删除 Key | ✅ | ❌ | ❌ | 仅紧急封禁 | ❌ |

## 五、核心业务流程

### 5.1 创建 API Key 流程

```text
客户管理员进入 /tenant/api-keys
  -> 点击“创建密钥”
  -> 填写名称、分组
  -> 可选：自定义密钥、IP 限制、额度限制、速率限制、有效期
  -> 系统校验企业状态、余额、权限、Key 数量上限
  -> 生成 Key 明文 + 哈希入库
  -> 弹窗展示明文，仅展示一次
  -> 客户复制保存
  -> Key 进入 active 状态，可立即调用
```

关键规则：

1. Key 前缀建议为 `ak_` 或 `sk-`，列表只显示前后片段，如 `sk-6ee...5959`。
2. Key 明文只在创建成功后展示一次；刷新、关闭后不可再次查看。
3. 客户可重置 Key，重置后旧 Key 立即失效。
4. 默认每企业最多 50 个 active Key；可由官方后台调整。
5. 创建 Key 时默认没有调用白名单限制，但建议 UI 提示配置 IP 白名单。

### 5.2 API 调用与转发流程

```text
客户系统 / 自动化 Skill 发起请求
  -> Header 携带 Authorization: Bearer <api_key>
  -> API Gateway 解析 Key
  -> 校验 Key 状态、租户状态、余额、IP、速率、额度、服务权限
  -> 生成 request_id，写入请求日志 pending
  -> 路由到实际服务：
       外呼接口 / 图像模型 / 第三方模型 / 我方内部服务
  -> 服务返回结果
  -> 网关按计费规则计算费用
  -> 写消费流水、扣企业余额、更新 Key 用量聚合
  -> 将响应返回给客户
```

### 5.3 外呼按次计费 MVP

```text
客户传入候选人电话、话术、面试时间约束等参数
  -> 我方外呼服务发起电话
  -> 外呼接口返回 call_id / status
  -> 本期按“发起成功一次”扣费
  -> 生成 api_billing_ledger 费用流水
  -> 使用记录显示：外呼服务、成功、¥0.50、call_id
```

默认建议：

| 项 | MVP 口径 |
|---|---|
| 计费单位 | 次 |
| 默认单价 | 后台可配置；会议举例 0.5 元 / 次 |
| 扣费时点 | 外呼发起成功后扣费 |
| 失败是否扣费 | 客户参数错误不扣；供应商已接单但对方未接听是否扣费需按供应商成本确认 |
| 余额不足 | 调用前拒绝，返回 `402 BALANCE_INSUFFICIENT` |

### 5.4 外呼按分钟计费预留

如果供应商可以稳定返回通话时长，则支持：

```text
外呼发起成功
  -> 先按预计最低费用预授权 / 冻结
  -> 回调返回实际通话时长
  -> 按分钟向上取整结算
  -> 多退少补，更新同一 request_id 的计费流水
```

MVP 可暂不做冻结，只保留字段：

| 字段 | 说明 |
|---|---|
| bill_unit | `request` / `minute` / `token` |
| duration_seconds | 外呼实际通话秒数 |
| charged_units | 实际计费单位数 |
| unit_price | 单价快照 |
| cost_amount | 对客扣费 |

### 5.5 余额不足流程

```text
调用前余额校验
  -> balance < 最小可扣金额
      -> 拒绝请求
      -> 返回 402
      -> 写入失败记录（不扣费）
      -> 触发低余额通知
  -> balance >= 最小可扣金额
      -> 继续调用
```

低余额提醒规则：

| 触发条件 | 通知对象 | 通知方式 |
|---|---|---|
| 余额低于 100 元 | 客户管理员、客户财务 | 站内通知 |
| 余额低于 50 元 | 客户管理员、客户财务、客户经理 | 站内 + 邮件 |
| 余额不足导致调用失败 | Key 所属分组管理员（V2）/ 客户管理员 | 站内 + 邮件 |
| 单日消费超过过去 7 日均值 3 倍 | 客户管理员 | 站内告警 |

| 状态 | 启用 / 禁用 / 已过期 / 余额不足保护 |
| 操作 | 使用密钥、导入到配置、禁用 / 启用、编辑、删除、重置 |

#### 6.1.4 行操作规则

| 操作 | 行为 |
|---|---|
| 使用密钥 | 打开“如何调用”抽屉，展示 Endpoint、Header、curl 示例 |
| 导入到配置 | 将 Key 写入我方 Skill / CCS / Claw 配置（如当前产品存在该配置入口）；无配置能力时隐藏 |
| 禁用 | 二次确认；禁用后所有请求立即返回 `401 KEY_DISABLED` |
| 启用 | 若未过期且企业状态正常可启用 |
| 编辑 | 可改名称、分组、限制策略；不可改 Key 明文 |
| 删除 | 软删除；保留历史调用记录和流水 |
| 重置 | 生成新 Key，旧 Key 立即失效；必须强确认 |

| 筛选 | 时间、Key、分组、服务、状态、错误码 |
| 列表字段 | 时间、request_id、Key 名称、服务、计费单位、数量、费用、状态、错误码、耗时 |

调用详情抽屉：

| 信息 | 说明 |

| 基础信息 | request_id、服务、Key、分组、调用 IP、时间、耗时 |
| 计费信息 | 单价快照、计费单位、数量、扣费金额、扣费后余额 |
| 技术信息 | HTTP 状态、平台错误码、供应商错误码、重试次数 |
| 安全处理 | 不展示敏感请求体；手机号等字段脱敏 |

### 6.4 A10.4 服务目录 / 模型页面

会议提到“两个页面”：一个 Key 页面，一个告诉客户当前能提供哪些模型 / 接口转发。MVP 可做简版服务目录。

建议路由：`/tenant/api-services`

## 七、业务规则

### 7.1 Key 生命周期

```text
active
  -> disabled（客户禁用 / 平台紧急封禁）
  -> expired（到期自动失效）
  -> deleted（客户删除，软删除）
  -> rotated（重置后旧 Key 进入 revoked）
```

| 状态 | 是否可调用 | 是否可编辑 | 是否保留流水 |
|---|---|---|---|
| active | ✅ | ✅ | ✅ |
| disabled | ❌ | ✅ | ✅ |
| expired | ❌ | 可改有效期后恢复 | ✅ |
| deleted | ❌ | ❌ | ✅ |
| revoked | ❌ | ❌ | ✅ |

### 7.2 扣费规则

| 规则 | 说明 |
|---|---|
| 单价快照 | 每次调用按调用时的服务价格生成快照，后续调价不影响历史流水 |
| 余额扣减 | 从 tenant API 余额或统一账户余额扣减，具体财务口径需评审确定 |
| 幂等 | 客户请求可携带 `Idempotency-Key`；同一 Key + 同一幂等键不重复扣费 |
| 失败扣费 | 按服务配置决定；默认客户参数错误不扣费，供应商已产生成本时可扣 |
| 重试扣费 | 平台内部重试不重复扣费；客户重复发起按新请求计费 |
| 退款 / 冲正 | 错扣通过负向 ledger 冲正，不修改原流水 |

### 7.3 限制规则

| 限制 | 规则 |
|---|---|
| 企业 Key 数量 | 默认最多 50 个 active Key |
| 单 Key 速率 | 默认无自定义限制，但平台有全局兜底限流 |
| 企业级 QPS | 官方后台可设置，避免单客户打爆服务 |
| IP 白名单 | 配置后只有白名单来源可调用 |
| 额度上限 | 支持按 Key 月度额度；超过后该 Key 拒绝调用 |
| 有效期 | 过期后自动拒绝调用 |

### 7.4 安全规则

1. Key 仅保存哈希，不保存明文。
2. Key 明文只展示一次；官方管理员和运维永远不可见。
3. 请求日志默认不记录完整请求体；敏感字段脱敏或不落库。
4. 手机号、身份证、候选人信息等外呼参数按客户业务数据处理，需脱敏展示。
5. Key 创建、禁用、删除、重置全部写审计日志。
6. 单日高额消费、异常 IP、失败率异常触发告警。

## 十、财务与计费联动

### 10.1 余额口径

推荐将 API 消费余额作为客户账单余额的一部分，但在 UI 上单独展示“API 可用余额 / 本月 API 消费”。

| 方案 | 优点 | 风险 |
|---|---|---|
| 共用客户账户余额 | 客户理解简单，充值一次可用于多类消费 | 席位 / token / API 不同成本口径需要财务清楚 |
| 独立 API 余额 | 费用边界清晰，适合按量接口 | 充值入口更多，客户体验更复杂 |

MVP 建议：**共用账户余额 + API 消费单独流水分类**。

### 10.2 充值入口

沿用主 PRD A7 账单与充值：

```text
客户管理员进入账单与充值
  -> 选择 API 充值 / 增值服务充值
  -> 对公转账
  -> 官方后台审核到账
  -> 企业余额增加
  -> API Key 调用可继续扣费
```

### 10.3 对账要求

1. 每条成功扣费必须能追溯到 request_id。
2. 每个 request_id 最多一条正向扣费流水，冲正另写负向流水。
3. 客户看到的“近 30 天费用”必须等于 ledger 聚合结果。
4. 财务导出的 API 消费流水包含：企业、Key、服务、时间、数量、单价、金额、余额。

## 十二、验收标准

### 12.1 功能验收

| 编号 | 标准 |
|---|---|
| AC-01 | 客户管理员可以在 `/tenant/api-keys` 创建 Key，并只在创建成功页看到一次明文 |
| AC-02 | Key 列表展示名称、脱敏 Key、分组、今日费用、近 30 天费用、速率限制、额度、过期时间、状态和操作 |
| AC-03 | 禁用 Key 后，使用该 Key 调用外呼接口返回 `KEY_DISABLED`，且不扣费 |
| AC-04 | 余额不足时，调用返回 `BALANCE_INSUFFICIENT`，写失败记录但不扣费 |
| AC-05 | 外呼发起成功后生成 request log 和 billing ledger，余额同步扣减 |
| AC-06 | 使用记录可按 Key、服务、状态、时间筛选 |
| AC-07 | 删除 Key 后历史使用记录和扣费流水仍可查询 |
| AC-08 | IP 白名单开启后，非白名单 IP 调用返回 `IP_NOT_ALLOWED` |
| AC-09 | 额度限制达到后，继续调用返回 `QUOTA_EXCEEDED` |
| AC-10 | 所有 Key 创建、禁用、删除、重置操作写审计日志 |

### 12.2 计费验收

| 编号 | 标准 |
|---|---|
| AC-B01 | 同一个 request_id 的幂等重放不重复扣费 |
| AC-B02 | 单价快照写入 ledger，服务调价不影响历史流水 |
| AC-B03 | 近 30 天费用 = ledger 按 Key 聚合金额 |
| AC-B04 | API 消费流水可被财务按企业、时间、服务导出 |
| AC-B05 | 错扣只能通过负向 ledger 冲正，不允许改原流水金额 |

### 12.3 安全验收

| 编号 | 标准 |
|---|---|
| AC-S01 | 数据库不保存 Key 明文 |
| AC-S02 | 官方管理员和运维无法查看 Key 明文 |
| AC-S03 | 请求日志不展示完整手机号等敏感参数 |
| AC-S04 | 所有管理接口强制校验 tenant_id，不能跨企业访问 |
| AC-S05 | Key 删除 / 重置必须二次确认 |

## 十四、风险与待决问题

### 14.1 风险

| 风险 | 影响 | 应对 |
|---|---|---|
| 外呼供应商计费口径不清 | 可能出现我方亏损或客户争议 | MVP 先按次，失败扣费规则必须评审 |
| 客户脚本失控高频调用 | 余额被快速打空，客户投诉 | 默认全局限流 + 额度限制 + 异常消费提醒 |
| Key 泄露 | 非授权调用产生费用 | IP 白名单、重置 Key、异常 IP 告警、审计 |
| 余额扣减与接口结果不一致 | 财务对账困难 | request_id 幂等 + ledger 不可变 + 冲正机制 |
| 下游服务不稳定 | 客户集成体验差 | 渠道状态页 + 错误码标准化 + 重试策略 |
| 主 PRD 原非目标冲突 | 范围理解混乱 | 本 PRD 评审通过后同步更新主 PRD 边界 |

### 14.2 待决问题

| 问题 | 负责人 | 建议决策 |
|---|---|---|
| API 消费是否共用客户账户余额 | 产品 + 财务 | MVP 共用余额，流水分类 |
| 外呼失败是否扣费 | 产品 + 后端 + 供应商接口负责人 | 按供应商是否产生实际成本决定 |
| 外呼默认对客单价 | 产品 + 财务 | 先配置化，会议举例 0.5 元 / 次 |
| Key 是否允许客户自定义 | 产品 + 安全 | MVP 可以保留开关，但默认关闭或灰度 |
| “导入到 CCS”是否已有真实目标配置 | 前端 + 交付 | 没有真实配置入口则隐藏 |
| 服务目录首批除外呼外还展示哪些模型 | 产品 + 商务 | 外呼 T0，其他手工配置后灰度 |
