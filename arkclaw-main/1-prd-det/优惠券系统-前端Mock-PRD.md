## 一、会议结论

本次会议确定：优惠不要通过修改套餐官方价格实现，而是通过给企业账户发放代金券 / 折扣券实现。

核心原因：

1. 套餐价格需要保持稳定，不能因为临时成交策略频繁改价。
2. 对价格敏感、历史谈判、跑马圈地类客户，可以通过优惠券让利。
3. 券用完、过期或不可用时，客户恢复按原价购买。
4. 第一版尽量上，方便首批客户成交时直接送券。

会议中明确的产品口径：

```text
管理员手动发券
  -> 券直接发到企业账户
  -> 客户购买时支付前抵扣
  -> 套餐原价不变
  -> 订单记录原价、优惠金额、实付金额
```

## 二、产品目标

### 2.1 本期目标

本 PRD 只服务前端 mock 原型，目标是让评审人员能在界面里完整看到：

| 目标 | 说明 |
|---|---|
| 后台能配置模板并发券 | 官方管理员可以配置券模板，并给客户发放优惠券或代金券；给销售管理员或普通销售只发放优惠券 |
| 企业能持券 | 企业详情能看到该企业名下优惠券和代金券、优惠范围、余额/使用状态、有效期和使用记录 |
| 支付前能抵扣 | 客户购买席位时，支付确认页展示可用券和抵扣结果 |
| 订单能对账 | 客户侧和后台订单详情都能看到原价、优惠金额、实付金额 |
| 不改套餐价 | `/tenant/purchase` 和 `/admin/plans` 仍展示官方原价 |

### 2.2 非目标

以下内容不进入本期前端 mock：

| 非目标 | 说明 |
|---|---|
| 真实后端扣减 | 不实现数据库事务、并发锁、支付回调真实扣券 |
| 公开券码 | 不做客户输入优惠码，不做扫码领券 |
| 自动发券 | 不做按活动、注册、代理等级自动发券 |

| 复杂营销券 | 不做平台活动券、公开券码、注册送券、用户自助领券 |
| 退款退券 | 仅在订单详情里预留退款后退券提示，不做完整链路 |
| 财务实收对账系统 | 只在订单金额构成中展示优惠，不扩展财务模块 |

## 四、核心流程

### 4.0 概念口径

| 概念 | 定义 | 例子 |
|---|---|---|
| 优惠券模板 | 无默认额度的折扣规则模板，决定优惠范围、折扣方式和默认有效期 | 整单 7 折优惠券模板、席位 8 折优惠券模板 |
| 代金券模板 | 带默认额度的折扣规则模板，决定优惠范围、折扣方式、默认有效期和默认代金券额度 | 席位 8 折代金券模板、CodingPlan 8 折代金券模板 |
| 优惠券 | 无可用额度的券，按券模板计算单笔订单优惠，使用一次后失效 | 给客户发放一张首单 7 折优惠券 |
| 代金券 | 有可用额度的券，按券模板计算优惠并扣减余额，可多次使用直到额度耗尽 | 客户获得 50,000 元席位 8 折代金券 |

优惠范围第一版只做三类：

| 优惠范围 | 说明 |
|---|---|
| 整单 | 订单内全部金额都可参与优惠 |
| 席位 | 只优惠 ArkClaw 席位订阅金额 |
| CodingPlan | 只优惠 CodingPlan 绑定服务金额 |

同一笔订单最多使用一张券。系统根据该券的优惠范围计算“可优惠金额”，再用折扣规则计算抵扣金额。优惠券不设余额，使用一次后状态变为 `used`；代金券抵扣金额不能超过剩余额度，余额用完后状态变为 `exhausted`。

### 4.1 后台手动发券

```text
官方管理员进入代金券与优惠券
  -> 先点击“新建券模板”，通过模板类型选择优惠券模板或代金券模板
  -> 再点击“发放券”
  -> 先选择券类型：优惠券 / 代金券
  -> 选择发放对象：客户 / 销售管理员 / 普通销售
  -> 选择同类型券模板，模板内包含优惠范围：整单 / 席位 / CodingPlan
  -> 填写券名称、折扣比例、有效期、备注；代金券额外填写总额度
  -> 发给客户：客户账户新增 active / pending 状态优惠券或代金券
  -> 发给销售管理员或普通销售：固定发放优惠券，按张数发放，不支持代金券
```

### 4.2 销售优惠券分发

```text
官方管理员 -> 销售管理员 -> 普通销售 -> 客户
官方管理员 -> 普通销售 -> 客户
官方管理员 -> 客户

客户最终收到的是企业账户优惠券或代金券，可在下单和支付时抵扣。
```

### 4.3 客户支付前抵扣

```text
客户进入购买席位页
  -> 选择套餐、席位数、周期
  -> 进入支付确认页
  -> 系统读取企业可用券
  -> 系统按优惠范围拆分订单金额
  -> 自动选中抵扣金额最高的一张券
  -> 展示订单原价、优惠金额、实付金额
  -> 客户生成扫码支付订单或对公转账订单
```

### 4.4 订单优惠记录

```text
订单生成后
  -> 订单详情保存本次 mock 金额构成
  -> 客户订单详情可见优惠明细
  -> 后台订单详情可见优惠明细
  -> 企业优惠券使用记录新增一条记录
```

## 五、券模板与券实例

### 5.1 第一版券模型

第一版采用“券模板 + 券实例”模型。

券模板分为优惠券模板和代金券模板。发放时先选择券类型，再只能绑定同类型模板：

```text
优惠券模板：保存优惠范围、折扣规则、默认有效期；不保存默认额度。
代金券模板：保存优惠范围、折扣规则、默认有效期和默认代金券额度。
优惠券实例：无总额度和余额；每笔订单按折扣比例计算优惠，使用一次后失效。
代金券实例：有总额度和剩余额度；每笔订单按折扣比例计算优惠，但累计优惠不能超过总额度。
```

示例：

| 字段 | 示例 |
|---|---|
| 券名称 | 首批客户 5 折权益券 |
| 折扣比例 | 50% |
| 总优惠额度 | ¥50,000 |
| 已用额度 | ¥10,000 |
| 剩余额度 | ¥40,000 |
| 有效期 | 2026-05-13 至 2026-12-31 |

### 5.2 抵扣公式

```text
理论优惠金额 = 订单原价 × (1 - 折扣比例)
本次优惠金额 = min(理论优惠金额, 券剩余额度, 订单原价)
实付金额 = 订单原价 - 本次优惠金额
```

如果使用“5 折券”，折扣比例记为 `50%`：

```text
订单原价 ¥200
理论优惠 ¥100
券剩余额度充足
本次优惠 ¥100
实付 ¥100
```

如果券余额不足：

```text
订单原价 ¥100,000
理论优惠 ¥50,000
券剩余额度 ¥30,000
本次优惠 ¥30,000
实付 ¥70,000
```

### 5.3 状态规则

| 状态 | 含义 | 页面表现 |
|---|---|---|
| 未生效 pending | 当前时间早于生效时间 | 灰色 Tag，不参与支付抵扣 |
| 可用 active | 在有效期内且有剩余额度 | 蓝色 Tag，可用于支付抵扣 |
| 已用完 exhausted | 剩余额度为 0 | 灰色 Tag，不可用 |
| 已过期 expired | 当前时间晚于失效时间 | 橙色 Tag，不可用 |
| 已停用 disabled | 管理员手动停用 | 红色 Tag，不可用 |

## 八、交互状态

### 8.1 可用券选择

当企业有多张可用券时，前端 mock 默认选择“本单优惠金额最高”的一张。

本期不做复杂选择器，最多展示：

```text
已自动选择：首批客户 5 折权益券
本单优惠：¥6,450
```

如需要展示多券，可在抽屉或下拉中列出，但一笔订单只能选一张。

### 8.2 不使用券

客户可勾选“本单不使用券”。

勾选后：

1. 本次优惠金额变为 0。
2. 实付金额恢复订单原价。
3. 金额构成中显示 `优惠抵扣：本单不使用`。

### 8.3 券不可用提示

| 场景 | 文案 |
|---|---|
| 未生效 | `该券尚未生效` |
| 已过期 | `该券已过期` |
| 已使用 | `该优惠券已使用` |
| 已用完 | `该代金券额度已用完` |
| 已停用 | `该券已被管理员停用` |
| 无券 | `暂无可用券，本单按套餐原价支付` |

### 8.4 实付为 0

前端 mock 允许出现实付为 0 的展示，但支付按钮文案调整为：

```text
确认订单
```

点击后模拟订单已支付。

## 九、验收标准

### 9.1 后台发券

| 编号 | 验收点 |
|---|---|
| AC-01 | 官方后台可以看到代金券与优惠券入口或企业详情优惠券 Tab |
| AC-02 | 管理员可以给企业发放额度折扣券 |
| AC-03 | 发券成功后，企业优惠券列表出现新券 |
| AC-04 | 券列表展示总额度、已用额度、剩余额度、有效期、状态 |
| AC-05 | 管理员可以查看券使用记录 |
| AC-06 | 管理员可以停用 active 状态优惠券 |

### 9.2 客户支付

| 编号 | 验收点 |
|---|---|
| AC-07 | 购买席位页仍展示套餐原价 |
| AC-08 | 支付确认页展示可用优惠券 |
| AC-09 | 支付确认页展示订单原价、优惠金额、实付金额 |
| AC-10 | 扫码支付金额等于实付金额 |
| AC-11 | 对公转账应转金额等于实付金额 |
| AC-12 | 无可用券时按原价支付 |
| AC-13 | 券余额不足时只抵扣剩余额度 |
| AC-14 | 勾选不使用优惠券后，实付金额恢复原价 |

### 9.3 订单展示

| 编号 | 验收点 |
|---|---|
| AC-15 | 客户订单详情展示优惠明细 |
| AC-16 | 后台订单详情展示优惠明细 |
| AC-17 | 使用优惠券的订单能跳转或关联到券使用记录 |
| AC-18 | 优惠金额不影响套餐原价展示 |

## 十、前端实现提示

### 10.1 视觉规范

遵循 `/1-prd/FRONTEND_GUIDELINES.md`：

1. 使用 Arco Design React 组件。
2. 搜索框使用 `Input.Search`。
3. 表格不使用竖向分隔线。
4. 使用控制台信息密度，不做营销式优惠视觉。
5. 不使用大面积装饰色、渐变背景、营销 hero。

### 10.2 推荐组件

| 场景 | 组件 |
|---|---|
| 优惠券列表 | `Table` |
| 发券表单 | `Drawer` + `Form` |
| 状态 | `Tag` |
| 已用额度 | `Progress` |
| 金额构成 | `Descriptions` 或轻量明细行 |
| 可用券卡片 | 白底面板，不做大促销卡片 |
| 停用确认 | `Modal.confirm` |

### 10.3 Mock 更新范围建议

如进入前端实现，建议优先改：

| 区域 | 说明 |
|---|---|
| `adminMockApi` | 增加优惠券 mock 数据、发券、停用、使用记录查询 |
| `AdminConsolePages` | 增加代金券与优惠券页、企业详情优惠券 Tab、订单优惠明细 |
| `BillingPage` / `PaymentPage` | 支付确认页展示可用券与实付金额 |
| `domain.ts` | 增加前端 mock 类型 |
| `AppLayout` / `appRoutes` | 增加后台优惠券路由和导航入口 |
