## 一、结论

本流程不再采用“每个客户成员手动输入激活码”的方案。

当前确定的业务闭环是：

```text
客户注册企业账号
  -> 销售归属确定
  -> 客户选择标准套餐或定制报价
  -> 客户对公转账
  -> 官方后台确认到账
  -> 系统生成开通任务
  -> 火山资源映射与统一登录配置
  -> 企业席位池开通
  -> 客户管理员进入席位管理
  -> 管理员邀请成员并分配席位
  -> 成员登录后自动启用席位
  -> 客户开始使用
  -> 销售佣金进入有效计算
```

“激活码 / 激活链接”在本流程中的定位是**企业订单激活入口**，只用于让客户管理员确认并激活企业订单，不用于每个成员逐个激活。

---

## 二、核心原则

1. 客户可以先开户注册，但开户注册不等于已开通能力。
2. Beta1 默认采用企业对公转账，不把微信 / 支付宝作为主购买路径。
3. 对公确认后进入“开通任务”，不直接承诺火山侧自动开通。
4. 企业订单只需要客户管理员激活一次，后续席位由管理员在席位管理中分配。
5. 客户成员不需要知道激活码，也不需要手动输入激活码；成员登录后根据管理员分配的席位自动启用能力。
6. 火山账号、空间、席位、Coding Plan 的映射是后端 / 交付依赖，不应暴露成客户操作步骤。
7. 佣金不以“付款已提交”为最终依据，建议以“企业席位池已开通并可使用”为 pending 佣金生成点。

## 三、角色与职责

| 角色 | 职责 |
|---|---|
| 客户管理员 | 注册企业、确认订单、对公转账、激活企业订单、邀请成员、分配席位 |
| 客户成员 | 接收邀请或通过统一登录进入工作台，按已分配席位使用能力 |
| 销售 / 客户经理 | 获客、绑定客户归属、推动报价和付款、跟进开通状态、查看佣金状态 |
| 官方后台 | 管理客户归属、创建定制订单、审核对公到账、生成和处理开通任务、处理异常 |
| 后端系统 | 支撑注册、订单、开通任务、火山资源映射、统一登录、席位池、佣金计算 |
| 火山 / 底层能力平台 | 提供账号、空间、席位、Coding Plan、SSO、能力开通相关接口或操作能力 |

## 四、主流程

### 4.1 注册与销售归属

```text
客户进入官网 / 注册页
  |
  v
填写手机号、企业信息、统一社会信用代码
  |
  v
系统校验企业是否已存在
  |
  v
创建我方 user / tenant / tenant_user
  |
  v
判断客户来源
  |-- 销售链接 / 注册邀请码 -> 自动绑定销售
  |-- 官网自然访问 / 咨询留资 -> 进入公共池或官方后台分配销售
  |
  v
形成销售归属
```

### 4.2 购买与对公确认

```text
客户选择标准套餐或提交定制需求
  |
  v
判断购买方式
  |-- 标准套餐 -> 系统生成标准订单
  |-- 定制报价 -> 销售报价 -> 官方后台创建定制订单 -> 客户确认
  |
  v
客户对公转账
  |
  v
客户上传付款凭证 / 填写汇款备注号
  |
  v
官方后台审核到账
  |-- 驳回 -> 客户补充凭证或重新付款
  |-- 通过 -> 订单确认已支付
```

### 4.3 企业开通与席位分配

```text
官方后台确认订单已支付
  |
  v
系统生成开通任务
  |
  v
映射火山资源 / 配置统一登录
  |
  v
判断开通是否成功
  |-- 失败 -> 人工处理 / 等待火山确认 / 重试
  |-- 成功 -> 开通企业席位池
  |
  v
客户管理员进入 /tenant/seats 席位管理
  |
  v
管理员邀请成员并分配席位
  |
  v
成员登录后自动启用席位
  |
  v
客户开始使用 ArkClaw 工作台
```

## 五、开通任务

### 5.1 触发条件

开通任务只在以下条件满足后生成：

| 条件 | 说明 |
|---|---|
| 订单已创建 | 标准订单或官方后台创建的定制订单 |
| 客户已确认订单 | 定制报价场景必须确认订单内容 |
| 对公到账已确认 | 财务 / 官方后台审核通过 |
| 销售归属已明确 | 自动归属或后台分配完成 |

### 5.2 开通任务状态

| 状态 | 含义 | 客户可见文案 |
|---|---|---|
| 待开通 | 已确认到账，等待系统处理 | 已收到付款，正在开通 |
| 开通中 | 正在映射火山资源、配置统一登录和席位池 | 正在为企业配置服务 |
| 待客户激活 | 企业级激活入口已生成，等待客户管理员确认 | 请管理员完成企业激活 |
| 已开通 | 企业席位池已可用 | 已开通，可分配席位 |
| 开通异常 | 火山映射、SSO 或席位池开通失败 | 开通遇到异常，客户经理会跟进 |

### 5.3 激活入口

激活入口可以是企业级激活链接，也可以是企业级激活码。

首版建议优先使用**激活链接**，激活码作为备用方案：

| 方式 | 使用场景 | 客户操作 |
|---|---|---|
| 企业级激活链接 | 客户管理员已登录或可通过站内消息触达 | 点击链接，确认激活企业订单 |
| 企业级激活码 | 邮件、线下合同、客户无法直接打开链接 | 客户管理员在企业激活页输入一次 |

激活入口只绑定订单和企业，不绑定单个成员。

## 六、席位管理

`/tenant/seats` 是企业开通后的核心承载页面。

该页面需要覆盖：

| 能力 | 说明 |
|---|---|
| 企业开通状态 | 展示待付款、待确认、开通中、可分配、开通异常 |
| 已购席位池 | 展示各套餐席位总数、已分配、剩余、到期时间 |
| 成员分配 | 管理员为成员分配轻量、标准、高级、旗舰等席位 |
| 批量操作 | 批量编辑成员配额或席位 |
| 扩容 / 变更 | 进入席位变更或联系销售报价 |
| 异常提示 | 开通失败、SSO 未配置、火山映射失败时提示联系客户经理 |

---

## 七、销售端关注点

销售端不再以“派发成员激活码”为核心，而是以“开通跟进”为核心。

销售需要看到：

| 信息 | 用途 |
|---|---|
| 客户归属 | 确认客户是否算自己 |
| 订单状态 | 是否已下单、是否已确认、是否已对公到账 |
| 开通状态 | 是否生成开通任务、是否开通中、是否异常 |
| 企业激活状态 | 客户管理员是否完成企业级激活 |
| 席位池状态 | 已购席位、已分配席位、剩余席位 |
| 佣金状态 | 未生成、pending、confirmed、paid、reverted |

销售可执行动作：

| 动作 | 边界 |
|---|---|
| 跟进客户补资料 / 补凭证 | 可以 |
| 提醒客户管理员完成企业激活 | 可以 |
| 查看开通异常并联系官方后台处理 | 可以 |
| 生成或修改火山资源映射 | 不可以 |
| 给每个成员派发激活码 | 不需要 |
| 直接修改佣金状态 | 不可以 |

---

## 八、官方后台关注点

官方后台需要补齐：

| 模块 | 功能点 |
|---|---|
| 订单与财务确认 | 标准订单、定制订单、对公凭证审核、驳回、挂起 |
| 开通任务 | paid 后自动生成任务，展示任务状态、失败原因、重试记录 |
| 火山资源映射 | 记录火山账号 / 空间 / 席位 / Coding Plan 的映射结果 |
| 统一登录配置 | 标记 SSO 是否完成、是否可供成员登录 |
| 企业激活入口 | 生成企业级激活链接 / 激活码，绑定订单和企业 |
| 佣金确认 | 企业开通可用后生成 pending 佣金，财务对账后确认 |

## 九、佣金规则建议

### 9.1 推荐触发点

```text
订单已对公确认
  |
  v
开通任务成功
  |
  v
企业席位池可用
  |
  v
生成 commission.status = pending
```

### 9.2 不建议的触发点

不建议在“客户提交付款凭证”或“订单刚变 paid 但尚未开通”时直接确认佣金。

原因：

- 对公凭证可能被驳回或挂起。
- 火山资源映射可能失败。
- 企业未开通时客户尚未获得可用服务。
- 销售端容易出现“已产生佣金但客户无法使用”的解释成本。

| P1 | 佣金触发点 | 需要和财务确认最终口径 |

## 十一、暂缓事项

以下能力不进入当前主流程：

- 每个成员一个激活码。
- 激活码池批量生成、过期重发、政策码、试用码。
- 客户侧复杂激活码输入页。
- 微信 / 支付宝即时支付。
- 火山 token 精细化控制的客户可见流程。
- 多销售协作分佣。
